## 1. M1：基础（表达式、打印、绑定）

### 1.1 语言面要点

- **顶层/语句级表达式**：整型、浮点、括号与完整二元表达式（含优先级 `*` 高于 `+` 等）。
- **绑定**：`x = expr`（flex）、`x : T := expr`（final + 类型注解）、复合赋值 `+=` 等。
- **打印**：`>_(expr, …)`；需支持 **i64 / f64 / string / bool** 等到 stdout。
- **运算符扩展**：`%`、`**`（幂）、比较、逻辑、`true`/`false`；整除与浮点除语义区分。

### 1.2 实现教训（里程碑中常见缺口）

1. **勿在 `INTEGER`/`DECIMAL` 语句分支只返回裸字面量**：顶层 `>_(1+2)` 等必须走 **`parse_expr`**（或等价二元表达式入口），否则链式算术在文件顶层无法解析。
2. **Lowering 忌占位**：`PrintAST`、`BlockAST`、`StringAST`、`ReturnAST` 若 stub 成 `SGConstInt(0)`，JIT 阶段会表现为「能编译、无输出或行为错」；应落到真实 **`SGPrint` / `SGBlock` / `SGConstString` / `SGReturn`** 等节点。
3. **打印 CodeGen**：`printf` 格式串与 LLVM 类型对齐（如 **`%lld`** 配 i64、**`%f`** 配 double、字符串用 **`i8*`**）；bool 宜格式化为字面值再打印。
4. **词法 `**`**：两个 `*` 须合并为 **`BINOP_POW`**（最大吞噬），避免被拆成两次乘法。
5. **CodeGen 补全**：`Binary_Mod`、`Binary_Pow`、比较与逻辑运算在 **TypeInfer** 与 **CodeGen** 两侧成对实现；**复合赋值**统一为 **load → 运算 → store** 到同一 **alloca**。
6. **`main` 返回值**：若语言语义要求「最后表达式值为退出码」，需 **`truncate_for_main_ret`**（或同类）把 void/指针/非 i32 规范到 **`i32`**，避免 LLVM verify 失败（后续里程碑会反复碰到）。

### 1.3 测试标签

- `ctest -L m1`：`tests/m1/t01`–`t20` 等。

## 2. M2：函数（定义、调用、返回）

### 2.1 语言面要点

- **定义**：`# name := (a: i32, …) => body`，可有显式返回类型；**`body`** 可为单表达式、`{ … }` 块，或带 **`<|`** 的显式返回。
- **调用**：`f(e1, e2)`；需支持 **多参数、浮点、嵌套定义、前后向引用**（同一顶层内互相调用）。
- **语义**：无 `<|` 时，**最后一条表达式的值** 作为函数返回值；**`main`** 同理可与 M1 的返回约定衔接。

### 2.2 实现教训

1. **形参进体**：对每个 LLVM **`Argument`** 做 **`alloca` + `store`**，再写入 **`mutable_variables`/`named_values`**，体中按变量名 **load**；若整段注释掉，会出现「解析正常、体内名字全未定义」。
2. **`FuncCallAST` → `SGCall`**： lowering 必须携带 **函数名 + 已 lowering 的实参列表**；CodeGen 侧 **`CreateCall`** 的 **function type** 与声明一致，返回值接到上层表达式。
3. **先签名、后体**：对 **相互递归**、**后定义先调用**，需在 **`ToStyioIR`**（或等价阶段）做 **预扫描**：先注册所有 **`SGFunc` 声明**，再 **`define_sgfunc_body`**，否则模块内找不到被调符号。
4. **`<|`**：解析为 yield/return 表达式，lowering 到 **`SGReturn`**；与 **块末隐式返回** 二选一或并存时，要保证 **基本块必须以 terminator 结束**，且不可达路径不重复 **`CreateRet`**。
5. **嵌套函数作用域**：进入内层 **`SGFunc`** 时 **保存并清空** `named_values`/`mutable_variables`（或栈式作用域），退出 **恢复**，避免内层参数污染外层或反之。
6. **类型推断与调用**：实参为 **`Id`** 时需在 **`typeInfer(FuncCallAST)`** 等路径能解析类型（否则实参个数统计错误、或污染 stdout），见 **M7 记录 7.3.7** 与 **`local_binding_types`**。

### 2.3 测试标签

- `ctest -L m2`：`tests/m2/t01`–`t10`（**T2.08** 含 `?=` 时依赖 M3，常作延伸）。

## 3. M3：控制流（match / 循环 / break / continue）

### 3.1 语言面要点

- **`?=`**：整型模式匹配 + 默认臂 `_`；可作语句或（配合 `<|`）作带值的表达式。
- **`[...] => { }`**：无限循环；**`[...] ?(cond) >> { }`**：条件循环；**`>>` 长度**可表达 continue 深度。
- **`^` 连续个数**：表达 break 跳出层数（多重重循环）。
- **集合迭代**：`collection >> #(var) => { }` 与 M5 的资源迭代共用 `>>` 记号，后续必须在**前缀类型**上分流。

### 3.2 实现教训

1. **LLVM `switch` + merge**：match 作为表达式时，各臂汇入 **phi**，需保证每条路径以 **同一 merge 块** 收口，且类型一致（分析器侧统一臂类型）。
2. **循环栈（loop stack）**：在 CodeGen 维护 `(exit_bb, continue_target_bb)` 栈；`BreakAST(n)` / `ContinueAST(n)` 按深度查表发 **`br`**，且**作用域出口**（正常走出块）必须与栈弹出顺序一致，否则 `break` 目标错位。
3. **词法与解析**：`^`、`>>`、`<|`、`?(` 等需在 **Tokenizer 最大吞噬** 下与现有 `>`、`-` 等区分，避免二义性。
4. **Visitor 全覆盖**：新增 AST 节点必须同步 **ToString / Analyzer / ToStyioIR / CodeGen** 链条，否则要到很后阶段才以 *NotImplemented* 暴露。
5. **`parse_arithmetic_tail` 与 `[`**：裸数字后的 `[` 易被误当成下一行列表或索引后缀（M7 对 bool/string 做了同类防护；M3 起就应意识到「字面量 + 下一 token `[`」的粘连风险）。

### 3.3 测试标签

- `ctest -L m3`：`tests/m3/t01`–`t10`。

## 4. M4：波算子与 `@`（缺失值代数）

### 4.1 语言面要点

- **`<~` / `~>`**：条件合并 vs 条件派发（真臂可有副作用，如 `>_(...)`）。
- **`@`**：未定义字面量；与算术、逻辑、`**|** fallback** 的传播规则需在类型与 codegen 两侧一致。
- **`expr | rhs`**：低优先级 fallback；仅当左侧「可视为 @」时取右侧。
- **选择器**：`x[?, cond]`、`x[?=, val]` 等与 `StateRef` 上的 `[<<, dep]`（M6）在 **`[` 后第一个 token** 上分流。

### 4.2 实现教训

1. **`~>` 落在哪一层语法**：若在 **`parse_arithmetic_tail_from_atom`** 中处理，则**所有**能接到 arithmetic tail 的表达式都能跟 `~>`；顶层若只走 `parse_fallback_expr` 而不接 tail，会出现「括号里可以、顶层不行」（M7 已用外层包 tail 修复同类问题）。
2. **`@` 的 LLVM 表示**：里程碑建议 **{value, is_undef}** 或与 **sentinel（如 i64 min）** 结合；一旦选定，**所有**可能为 `@` 的二元运算前都要 **guard**，避免把 sentinel 当普通数参与运算。
3. **`<~` vs `~>` codegen**：合并多用 **select**；派发多用 **条件分支 + 两个基本块**，以便真臂执行打印等副作用。
4. **`|`**：与逻辑或 `||`、位或、管道等符号在词法与优先级上要严格区分（`parse_fallback_expr` 内层循环专管 `|`）。

### 4.3 测试标签

- `ctest -L m4`：`tests/m4/t01`–`t15` 等。

## 5. M5：资源与 I/O

### 5.1 语言面要点

- **`f <- @file{...}` / `@{...}`**：句柄获取；**`f >> #(line) =>`**：按行读。
- **`expr >> @file{...}`**：写入资源（与迭代 **`>>`** 二义，靠 **`>>` 后是否为资源原子** 区分）。
- **`expr -> @file{...}`**：重定向写（语义上常与「单行写入 + 是否追加换行」相关）。
- **RAII**：块结束、break、return 路径上都要 **close** 同一 alloca 对应的句柄，避免重复 close 或遗漏（常按 **alloca 去重** 或 **作用域栈** 实现）。

### 5.2 实现教训

1. **`>>` 双义性**：解析与 **parse_expr_postfix** 中 **`>>` 后接 `@`** 分支必须稳定；遗漏会导致 IR 完全偏离（例如把写文件当成返回值）。
2. **T5.02 语义**：**`"Hello" >> @file{path}`** 表示把 **字符串写入文件**（数据流方向：表达式 → 文件），不是「把文件写进字符串」；解析与 lowering 必须以 **左值为数据、右值为资源** 为准。
3. **`name << @file`**：语句级 **`<<`** 与 **`parse_return` 的 <<** 易混淆；需在 **`NAME` 探测** 阶段单独识别 **资源写**（M7 会话中修复）。
4. **打开模式**：多次迭代向同一文件写宜用 **`"ab"`** 追加；与「每脉冲一行」类测试一致。
5. **失败策略**：打开失败时 **stderr + 非零退出** 优于静默或坏句柄；注意 **不要往 stdout 打诊断**，以免破坏 `cmp` golden。
6. **运行时 API 契约**：`read_line` 返回的指针生命周期若指向 **复用缓冲**，任何 **连续两次 read 且均保留指针** 的场景（如 **zip 两路读**）必须 **复制或双缓冲**（M7 在 `styio_file_read_line` 中采用双缓冲）。

### 5.3 测试标签

- `ctest -L m5`：`tests/m5/t01`–`t08` 等。

## 6. M6：状态容器、脉冲与内建

### 6.1 语言面要点

- **`@[init 或窗口](update)`**：状态声明；**`$var`**：帧内读（经快照）；**`$var[<<, n]`**：历史探针。
- **帧锁（pulse）**：同一迭代体内多次读 `$var` 必须一致 → **先拷 ledger → snap，体中只读 snap，结束再写回**。
- **内建 `[avg, n]` / `[max, n]`**：滑动窗口；冷启动阶段可产生 `@`（与 M4 语义衔接）。

### 6.2 实现教训（架构权重最高）

1. **单一 ledger**：所有 `@[...]` 布局进 **一块连续字节**（或固定 alloca），用 **offset + GEP** 访问，满足「零分配、可预测」目标。
2. **Pulse 计划**：含状态的块需 **pulse_plan**（哪些 slot 在体前 reload、体后 commit）；与 **for-each / file-line-iter / zip** 等多入口循环统一挂钩，避免只处理一种循环。
3. **快照与 shadow**：迭代参数、文件行变量与 **`$snapshot`** 类变量在同一 pulse 的读序要一致；**ToStyioIR** 需标记 **snapshot 变量**，**CodeGen** 在体入口 **reload**。
4. **历史下标**：环缓 **`(cursor - n + cap) % cap`** 要与 **count 未满月** 的边界一致，避免读未初始化槽。
5. **跨函数与匿名 ledger**：内联或跨 `process(d)` 调用时，ledger 布局与 **IR  lowering** 必须同一套分析 pass 结果，否则 offset 漂移。
6. **与 M7 的衔接**：zip 双流、快照 `<< @file`、多 `@file` 单例 handle 都会在 **pulse 区域** 与 **RAII** 上叠加，改 M7 时优先跑 **m6** 全量。

### 6.3 测试标签

- `ctest -L m6`：`tests/m6/t01`–`t10`。

## 7. M7：多流与会话内具体问题（详细记录）

### 7.1 目标与范围

- **语言特性**：流 zip（`&`）、快照 `@[v] << @file{...}`、即时拉取 `(<< @file{...})`、字符串与整型在打印与拼接中的一致行为、同路径 `@file{...}` 的 driver 单例、测试 **T7.01–T7.10**（`tests/m7/t01`–`t10`）。
- **约束**：与 **M1–M6** 测试一并回归。

### 7.2 会话前已存在的大块实现（摘要）

- **分析器 / IR**：`TypeInfer.cpp`、`ToStyioIR.cpp`（含 `StreamZip`、`SnapshotDecl`、`InstantPull`、`StateRef` 快照、`Binary_Add` 字符串推断等）。
- **代码生成**：`CodeGenG.cpp`、`CodeGenVisitor.hpp`（`SGStreamZip`、快照、单例 handle、`styio_strcat_ab` 与 promote/newline 等）。
- **解析器**：zip 第二边用 `parse_fallback_expr`、`TOK_DOLLAR` 在 `parse_binop_item` 中等。
- **运行时**：`styio_file_rewind`、写文件 **`"ab"`** 追加模式等。

### 7.3 本次新发现并修复的问题

#### 7.3.1 解析：`n + " " + s`（m7 t01 / t02）

**现象**：`line = n + " " + s` 在 `s` 处报 *No Statement Starts With This*。

**根因**：`parse_binop_item` 在消费 `STRING` / `INTEGER` / `DECIMAL` 以及字面量 `true`/`false` 后，未接上 `parse_arithmetic_tail_from_atom`，RHS 无法继续链接 `+`。

**修复**：在 `parse_binop_item` 中为上述分支补上 `parse_arithmetic_tail_from_atom`。

#### 7.3.2 解析：`(cond) ~> … | @`（m7 t08）

**现象**：`StyioNotImplemented: Unknown Expression`。

**根因之一**：`gap < -5` 中 RHS 以 `-` 开头，`parse_arithmetic_expr` 原先无一元 `-`/`+`。

**修复**：在 `parse_arithmetic_expr` 增加 `TOK_MINUS` / `TOK_PLUS` 一元情形（减号用 `0 - inner`）。

**根因之二**：`parse_expr` 仅 `parse_expr_postfix(parse_fallback_expr(...))`，`~>` 只在 `parse_arithmetic_tail_from_atom` 中处理；顶层不在 `||`/`&&` 之后接 arithmetic tail。

**修复**：`parse_expr` 改为  
`parse_expr_postfix(context, parse_arithmetic_tail_from_atom(context, parse_fallback_expr(context)))`。

#### 7.3.3 解析：`result << @file{…}`（m7 t10）

**现象**：LLVM verify：`ret ptr @styio_str.x` 与 `main` 的 `i32` 不匹配。

**根因**：`result << …` 被拆成「`result`」+「`<<` 作 `parse_return`」，IR 错误。

**修复**：在 `parse_stmt_or_expr` 的 `NAME` 探测分支中，若下一 token 为 `EXTRACTOR`（`<<`），构造 `ResourceWriteAST`。

#### 7.3.4 解析：`gap = a - b` 与下一行 `(` 被粘成函数调用（m7 t08）

**根因**：`parse_name_and_following_unsafe` 在 `b` 后 `context.skip()` 跨过换行，把下一行 `(` 当成 `b(...)`。

**修复**：`skip_spaces_no_linebreak()` + 在名称后续分支使用。

#### 7.3.5 解析：`result = true` 下一行 `[1,2,3]`（m7 t06）

**根因**：`parse_arithmetic_tail` 中 `[` 把 `true` 当作可下标对象。

**修复**：对 **Bool**、**String** 字面量与 Integer/Float 一样在 `[` 分支 **break**。

#### 7.3.6 代码生成 + 运行时：zip 双文件 `0,0,0`（m7 t05）

**根因 A**：文件边未按数值元素做 `styio_cstr_to_i64`。

**修复**：`SGStreamZip` 的 list–file / file–list / file–file 在 `!*_elem_string` 时对行指针转换并写入 `i64` slot。

**根因 B**：`styio_file_read_line` 单缓冲，连续两次 read 指针别名。

**修复**：双缓冲交替。

#### 7.3.7 分析器：`double_it(line)` 与 stdout（m5 t08）

**根因**：`FuncCallAST` 实参未处理 `Id`，且 `std::cout` 诊断破坏 `cmp`。

**修复**：`Id` 从 `local_binding_types` 推断（缺省 `i64`）；去掉 stdout 诊断。

## 8. 跨里程碑共性经验（合并小结）

1. **表达式入口一致性**：`parse_expr` 与 `parse_arithmetic_tail`、`parse_fallback_expr` 的包裹顺序决定 **`~>`、`|`、`+`** 等能否在「括号外」出现；改一处要想到 **顶层与子表达式是否对称**。
2. **`skip()` 与换行**：全空白跳过易把 **下一语句** 粘进当前表达式（调用括号、列表头 `[`）；需 **按语法位置** 选用「是否跨行」的跳过策略。
3. **返回缓冲与别名**：C 接口返回 **内部静态/thread_local 指针** 时，调用方若保留多个结果，必须 **拷贝或轮换缓冲**。
4. **测试与 I/O**：依赖 **`cmp -s`** 的测试要求 **编译器/分析器不写 stdout**；错误信息用 **stderr** 或日志开关。
5. **`>>` / `<<` 家族**：资源迭代、写入、`<<` 赋值/返回、快照 **`<< @file`** 等多条语法共享记号，**解析分层**（stmt 名探测 vs expr postfix）要逐项对齐。
6. **M6 pulse + M5 RAII + M7 多流**：动循环体或文件句柄时，同时检查 **pulse 提交**、**close 去重**、**单例 path**，避免「单测过、组合挂」。

## 10. 资源拓扑目标设计（2026-03 对话沉淀）

一次独立的设计讨论将 **`@`** 的三种角色（缺失值 / 外部资源锚点 / 状态容器）、**`[|n|]`** 有界环缓、**顶层 `@name : Type := { 驱动流 }`**、以及 **影子写入必须用 `expr -> $name` 而非 `$name = expr`** 写入了 **`docs/design/Styio-Resource-Topology.md`**，并在 **`docs/design/Styio-EBNF.md` Appendix B** 中给出目标 EBNF 摘要。

**当前编译器**仍以 **M6 的 `@[…](…)`** 等实现为准；完整迁移需新词法 **`[|` / `|`]**、顶层 **`ResourceDecl`** 语义与测试迁移，见拓扑文档 **§9 实现状态**。

## 11. Topology v2 / M8 — 开发会话记录（2026-03-28）

本节记录 **实现步骤与原因**（符合 `../plans/Resource-Topology-v2-Implementation-Plan.md` §5 与 `../specs/DOCUMENTATION-POLICY.md` §0）。

### 11.1 目标（本批次）

- **P1 词法：** 引入 **`[|`**、**`|]`**，与单字符 **`[`**、**`|`** 区分，避免与 M3/M6 的 `[` 流语法冲突（**最长匹配 / 成对界符** 策略，见 Topology §3、`../design/Styio-EBNF.md` B.4）。
- **可扩展解析入口：** 集中 **`parse_styio_type()`**，后续可在同一函数增加「元素类型、资源引用类型」等分支，而 **不** 在每条语句里复制 `:` 后的类型解析逻辑（**单点扩展**）。
- **AST：** `TypeAST::CreateBoundedRingBuffer(cap_digits)` 使用 `StyioDataTypeOption::Defined` + 规范名前缀 **`bounded_ring:`**，保留 **容量信息** 供后续 analyzer/ledger；**暂**复用现有 `SGType`→LLVM 对 `Defined` 的 **i64 回退**，使 `x : [|2|] := 1` 与 `>_(1)` 能 **端到端跑通**（**bootstrap 语义**：变量仍是单槽 i64，**非**真实环缓 — 避免本批次卷入 CodeGen 大改）。

### 11.5 环缓 CodeGen 替换 i64 Bootstrap（2026-03-28）

**动因：** 用户确认 **允许暂时破坏** 单 i64 占位语义，要求 **给出调整方案** 并落地真实 **有界槽**。

**实现要点：**

- 新增 [`src/StyioUtil/BoundedType.hpp`](../../src/StyioUtil/BoundedType.hpp)：`styio_bounded_ring_capacity(StyioDataType)` 解析 `bounded_ring:n`（**单点扩展**，供 IR / 类型共用）。
- **`GetTypeG.cpp`：** `SGType` → `ArrayType i64 x n`；`SGVar` 对环返回 **逻辑 i64**（脉冲值）；**`SGFinalBind`** 的 `toLLVMType` 改为走 **`var->var_type`**，保证 alloca 为 **数组** 而非被 `SGVar` 收成标量。
- **`CodeGenG.cpp`：** `SGFinalBind` 环分支在 **函数入口** `prealloc` 分配 `[n x i64]` 与 **head**；初值写入 `slot[0]`，`head=1`；**不**写入 `named_values`。`SGResId` 优先走 **环读**：`(head-1) mod n`，`head==0` 则 **0**。
- **`define_sgfunc_body`：** `bounded_ring_*` 与 `mutable_variables` 一并 save/restore，避免嵌套函数污染主程序环表。
- **文档：** [`../plans/BoundedRing-Codegen-Adjustment.md`](../plans/BoundedRing-Codegen-Adjustment.md)（破坏面、测试策略、回滚）；`TEST-CATALOG` 增加 `m8_t02_bounded_read`。

**已知缺口（与调整方案一致）：** Flex 覆盖、`#(x:[|n|])` 参数、多脉冲 push、`->` 写入 — 见调整文档 §2、§6。

### 11.6 Final 之后禁止 Flex（同名 `=`）（2026-03-28）

**规则：** `x : T := …` 将 `x` 记入 `StyioAnalyzer::fixed_assignment_names_`；`typeInfer(FlexBindAST*)` 若命中该集合则 **`StyioSyntaxError`**。

**原因：** 与语言「fixed / final」语义一致，并避免环绑等存储形态与 **另一套 mutable alloca**  silently 分叉（见 `../plans/BoundedRing-Codegen-Adjustment.md`）。

**作用域：** `IteratorAST` / `StreamZipAST` 与 `local_binding_types` 同样 **save/restore** `fixed_assignment_names_`，保证 **体内** 仍禁止 flex 外层 final；**体内新 final** 不会泄漏到外层。

**测试：** `tests/milestones/m8/e01`–`e10`（`ctest -L m8_semantic`）；正例 `t14_flex_other_var_after_final_ok`（对 **另一变量** flex 仍合法）。
