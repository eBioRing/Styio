## Checkpoint（Test Quality / Delivery：checkpoint-health build-dir fallback fix）

### 当前状态

- 在完整执行 `./scripts/delivery-gate.sh --mode checkpoint --with-asan --with-fuzz` 时，暴露了 `scripts/checkpoint-health.sh` 的真实缺陷：
  - `configure_build_dir_latest()` 在命令替换里执行 `cmake -S . -B ...`
  - `cmake` 的 stdout 被一起捕获进 `BUILD_DIR`
  - 当 `build/` 失败并回退到 `build-codex/` 时，后续 `cmake --build "$BUILD_DIR"` 实际拿到的是“配置日志 + build-codex”的拼接字符串，导致目录名损坏
- 已修复为：
  - `cmake` 配置输出统一走 stderr
  - 函数 stdout 只保留最终目录名
  - fallback configure 若失败，函数直接返回失败，不再伪造目录值
- [../teams/TEST-QUALITY-RUNBOOK.md](../teams/TEST-QUALITY-RUNBOOK.md) 已补充这条恢复路径约束，避免后续再破坏 build-dir handoff。
- 随后已继续补齐三条阻塞：
  1. `StyioSamples.BubbleSortListInput` 改为引用仓库真实存在的 `example/algorithms/bubble_sort.styio`
  2. `fuzz_smoke` 改成独立 corpus-replay smoke binaries，保留 libFuzzer 目标给手动/夜间深跑，不再把 PR 门禁绑在当前环境的 libFuzzer 启动差异上
  3. 顶层 `CMakeLists.txt` 与 `tests/CMakeLists.txt` 现在会优先复用本地 `tree_sitter_runtime` / `googletest` source checkout，`build-asan-ubsan` 可在离线环境下完成配置并执行目标 ASan 腿
- 重新核验后，`BubbleSortListInput`、`fuzz_smoke` 和 checkpoint-health 所需的 ASan 定向用例都已转绿。

### 下一步

1. 继续把完整 `delivery-gate --with-asan --with-fuzz` 保持在绿色基线，避免后续改动把 corpus-replay smoke 又拉回到直接执行 libFuzzer main。
2. 若后续要恢复 PR 级直接 libFuzzer smoke，必须先证明当前环境下入口启动行为稳定，再考虑替换现有 replay gate。
3. 维持本地 source reuse 路径，新增 FetchContent 依赖时同步提供离线恢复策略。

### 复现命令

```bash
./scripts/delivery-gate.sh --mode checkpoint --with-asan --with-fuzz
```

### 风险

1. `fuzz_smoke` 现在更偏“seed replay stability gate”而不是直接 libFuzzer runtime smoke；更深的随机探索仍依赖手动/夜间 fuzz 入口。
2. `checkpoint-health` 仍然会主动 `cmake -S . -B ...`；未来若引入新的远端依赖，必须同步补本地 source reuse 或显式 skip 策略。
